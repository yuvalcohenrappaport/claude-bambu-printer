**Section 1: Geometry Risk Patterns**
A checklist Claude uses to analyze .scad source code for printability risks. Organize into three tiers:

HIGH RISK (warn BEFORE generating -- per user decision):
- sphere() or complex curves as primary geometry (organic shapes)
- rotate() with angles creating overhangs > 60 degrees
- Very thin features: any dimension < 1mm
- Unsupported horizontal surfaces spanning > 50mm
- Complex boolean chains: 4+ nested difference()/intersection()
- hull() connecting distant points (unpredictable geometry)

MEDIUM RISK (note AFTER generating -- per user decision):
- rotate() creating 45-60 degree overhangs
- Bridge spans between 30-50mm
- Interlocking parts with tight tolerances (< 0.3mm clearance)
- Features smaller than 2mm in any dimension
- hull() between nearby points

LOW RISK (no special note needed):
- Axis-aligned primitives: cube(), cylinder() with no rotation
- Simple difference() operations (holes, cutouts)
- Chamfers and fillets at 45 degrees
- Features > 3mm in all dimensions
- Standard patterns: boxes, brackets, holders, stands

**Section 2: Confidence Language Templates**
Natural language examples Claude should use (NO percentages, NO tier labels -- per user decision):
- Low risk: "Straightforward geometry, this should print cleanly."
- Medium risk: "This has some overhangs around 50 degrees -- it should work but you may want to check the preview. Supports might help if the bridging looks rough."
- High risk with alternative: "Organic curves like this are tricky to get right in OpenSCAD -- the mesh may have rough spots. I can try it, or I can offer a simpler geometric version that'll definitely print well. Which do you prefer?"
- High risk explanation: "The overhang here is about 70 degrees from horizontal. Without supports, the filament will sag because there's nothing underneath to bond to."

Include 5-6 more examples covering: thin walls, long bridges, complex booleans, tight tolerances, and large flat unsupported spans.

**Section 3: Messiness Detection**
Heuristics for when to suggest regenerating from scratch (per user decision: after 5+ iterations):
- Version count >= 5 (tracked by backup file numbering)

- Suggested message template: "This model has gone through several iterations and the code is getting complex. Want me to regenerate it from scratch based on what we have now? I'll keep the current version as a backup."

**Add Step 9: Modify Existing Model**

This step is triggered when the user asks to change a previously generated model. Instructions:

1. **Locate the model**: Check conversation context for recent models. If starting fresh, browse `~/3d-prints/` directories to find the .scad file.

2. **Create versioned backup before ANY edit** (per user decision):
   ```bash
   OUTPUT_DIR="$HOME/3d-prints/<project-name>"
   NEXT_VER=1
   while [ -f "$OUTPUT_DIR/model_v${NEXT_VER}.scad" ]; do
       NEXT_VER=$((NEXT_VER + 1))
   done
   cp "$OUTPUT_DIR/model.scad" "$OUTPUT_DIR/model_v${NEXT_VER}.scad"
   ```

3. **Read and understand the .scad file**: Identify parametric variables, module structure, and how dimensions relate. This understanding is needed for both modification and scaling.

4. **Apply the requested modification**: Edit model.scad in place. For additions (holes, screw mounts, features), add new modules. For changes (thicker walls, different dimensions), modify parametric variables.

5. **Show changes as a summary diff** (per user decision -- do NOT auto-render):
   ```
   Changes made to model.scad:
     - Added: screw_hole_d = 5mm (M5 screw holes)
     - Added: module screw_holes() at lines 45-55
     - Modified: height 40mm -> 50mm

   Want me to render this?
   ```

6. **Only render after user confirms**. Use the same render/export commands from Steps 5-6.

7. **Check messiness** (per user decision): Reference @reference/printability-checklist.md messiness detection section. If version count >= 5 AND code quality signals fire, suggest regeneration.

**Add Step 10: Scale/Resize Model**

This step is triggered when the user asks to resize, scale, or change dimensions.

1. **Parse the scaling request**: Accept both relative ("20% bigger", "half the height") and absolute ("set width to 10cm", "make it 150mm tall") inputs. Parse which axes are affected.

2. **Preferred approach -- modify parametric variables** (per user decision):
   - Read the .scad file and identify dimension variables (width, depth, height, etc.) and wall/structural variables (wall, clearance, tolerance, etc.)
   - For uniform scaling ("20% bigger"): multiply ALL dimension variables by the scale factor. DO NOT modify wall thickness, clearance, or tolerance variables -- these must stay at their designed values.
   - For per-axis scaling ("10cm wider"): modify only the relevant axis variable(s). Leave wall and other axes unchanged.
   - If dimensions are hardcoded in the body (magic numbers), refactor them to parametric variables first, then scale.

3. **Fallback -- scale() transform** (per user decision: only if parametric approach is impossible):
   - If the .scad file has no clear parametric variables (e.g., imported from external source, heavily procedural), explain this to the user.
   - Offer two options: (a) refactor to parametric first, then scale, or (b) use scale() with the caveat that wall thickness will change.
   - If using scale(): wrap the entire model in `scale([sx, sy, sz]) { ... }` and warn about wall thickness changes.

4. **Auto-fix printability after scaling** (per user decision):
   - After scaling, check if any wall thickness fell below minimum (1.2mm PLA/PETG, 1.6mm TPU).
   - Check if features became too small (< 1mm).
   - If issues found, auto-fix and tell the user: "I adjusted wall thickness from 1.0mm back to 1.2mm (PLA minimum) after scaling."

5. **Create backup and show diff** (same as Step 9 -- backup, show changes, ask before rendering).
